# 基于内容推荐的校园社交平台设计与实现说明

## 1. 项目概述

本系统是一个面向校园场景的社交平台，围绕“内容发布、内容浏览、互动交流、后台治理、个性化推荐”五个核心目标展开设计与实现。系统既满足普通用户在校园内进行动态发布、话题浏览、关注互动、消息接收等需求，也满足管理员对内容审核、用户治理、标签维护、公告发布、推荐策略配置和数据统计分析等管理需求。

与传统校园论坛相比，本系统的重点不只是提供内容承载能力，而是通过推荐机制提升内容分发效率，使用户能够在首页优先看到与自身兴趣、行为偏好和关注关系更相关的校园内容，从而提高内容曝光质量、点击率和互动活跃度。

本项目采用前后端分离架构实现。前端负责用户界面展示、交互流程组织与页面状态管理；后端负责业务逻辑、权限校验、推荐计算、数据持久化和接口服务输出。数据库使用 MySQL，推荐、行为、关系和内容数据统一沉淀于 `school_social` 库中。

## 2. 系统开发环境与技术选型

### 2.1 后端技术栈

- 开发框架：Spring Boot 2.7.18
- 持久层框架：MyBatis
- 数据库：MySQL 8.x
- 权限认证：JWT + 自定义登录鉴权
- 参数校验：Spring Validation
- 构建工具：Maven
- JDK 版本：Java 8

后端服务配置在 `school_service` 模块中，端口为 `8080`，数据库连接地址为：

```yaml
jdbc:mysql://localhost:3306/school_social?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
```

### 2.2 前端技术栈

- 前端框架：Vue 3
- 构建工具：Vite
- 路由管理：Vue Router
- 状态与页面交互：Composition API
- 组件库：Element Plus
- 图标库：@element-plus/icons-vue

前端项目位于 `user_view` 模块，采用现代化后台与用户端一体化单页应用模式，实现用户端页面与管理员端页面的统一接入。

## 3. 系统总体架构设计

本系统采用典型的三层架构设计。

### 3.1 表现层

表现层由 Vue 3 前端构成，主要页面包括登录注册页、用户首页、个人中心、内容发布页、内容详情页、互动中心，以及管理员总览、统计、审核、举报、运营配置等页面。前端通过 `fetch` 封装的接口请求与后端通信，基于 JWT Token 完成身份识别。

### 3.2 业务层

业务层由 Spring Boot 控制器与服务组成。控制器负责接收请求和返回统一响应，服务层负责业务计算与流程编排。其中推荐服务是本系统的重要业务核心，承担内容召回、排序打分、曝光记录、点击反馈处理和推荐效果统计等职责。

### 3.3 数据层

数据层由 MyBatis Mapper 和 MySQL 数据库组成。系统将用户、内容、互动、关注关系、行为日志、推荐日志、审核日志、标签与配置项等数据统一存储，实现内容平台与推荐系统的数据闭环。

## 4. 系统功能模块设计

## 4.1 用户端功能模块

### 4.1.1 用户注册与登录

用户可通过注册页完成账户创建，系统支持用户名、邮箱、手机号等信息录入。登录成功后，前端将认证信息保存至本地存储，后续请求通过 Token 自动携带身份凭证。系统区分普通用户和管理员角色，管理员可进入后台管理页面。

### 4.1.2 首页内容浏览

首页是用户端核心页面，采用内容流式浏览设计，支持以下几类内容入口：

- 个性推荐流
- 关注流
- 热议流
- 最新发布流
- 话题内容流
- 关键词与话题组合搜索

首页还包含焦点内容卡片、侧边推荐内容、热门话题、未读消息数、快捷入口等模块，突出“浏览优先、互动跟进”的校园社交使用方式。

### 4.1.3 内容发布与编辑

用户可以发布校园动态，录入标题、正文、内容类型、可见性、位置、学院、标签和媒体资源等信息。系统支持内容草稿保存、内容编辑、内容删除和发布后审核流转。

### 4.1.4 内容详情与关联内容

用户点击内容后进入详情页，可查看正文、标签、媒体、互动统计和关联推荐内容。系统在详情页访问时会自动累加浏览量，并记录用户行为数据，为后续推荐计算提供依据。

### 4.1.5 互动社交功能

平台围绕校园内容提供基础互动能力，包括：

- 点赞
- 评论
- 收藏
- 关注用户
- 分享
- 举报
- 通知提醒

这些互动行为一方面提升社区活跃度，另一方面也为推荐系统提供了真实的行为反馈信号。

### 4.1.6 个人中心

个人中心用于展示和维护用户资料，包括头像、简介、学校、学院、年级、个人标签偏好、已发布内容、互动资产等信息，支持用户维护自己的校园身份与兴趣画像。

## 4.2 管理员端功能模块

### 4.2.1 数据统计看板

管理员可在统计页查看用户总数、新增用户、活跃用户、新增内容、互动量、留存率等数据，并按时间范围查看趋势变化。这部分功能用于反映平台运行状态和增长质量。

### 4.2.2 控制总览

控制总览页对内容审核、用户治理、公告管理和推荐策略进行统一摘要展示，帮助管理员快速掌握平台运行状况。

### 4.2.3 内容审核

管理员对用户发布内容进行审核，审核结果写入审核日志，并控制内容是否可进入正式内容池。只有审核通过的内容才会进入推荐计算和前台公开展示。

### 4.2.4 举报处理

系统支持用户对违规内容或异常行为进行举报。管理员在后台集中查看举报记录、处理状态和处理说明，形成完整的风控闭环。

### 4.2.5 运营配置

运营配置模块包括以下几个子功能：

- 内容运营：对已通过内容设置置顶或精选
- 用户治理：对异常用户进行封禁或解禁
- 标签体系：维护平台话题与兴趣标签
- 公告中心：发布或编辑站内公告
- 推荐策略：配置推荐开关与各类权重

该模块是后台运营能力的集中体现，也是本系统实现“内容治理 + 分发优化”的关键入口。

## 5. 数据库设计说明

本系统围绕校园社交和推荐功能建立了较完整的数据模型，主要数据表可分为以下几类。

### 5.1 用户与权限类

- `users`：用户基础信息
- `roles`：角色信息
- `user_roles`：用户角色关联
- `user_tags`：用户兴趣标签及权重

### 5.2 内容类

- `posts`：帖子主表
- `post_media`：帖子媒体资源
- `post_tags`：帖子与标签关联
- `tags`：平台标签
- `announcements`：平台公告

### 5.3 互动类

- `likes`：点赞记录
- `comments`：评论记录
- `favorites`：收藏记录
- `follows`：关注关系
- `notifications`：通知消息
- `reports`：举报记录
- `audit_logs`：审核日志

### 5.4 推荐与分析类

- `behavior_logs`：用户行为日志
- `recommendation_logs`：推荐曝光与点击日志
- `recommend_config`：推荐参数配置

其中，`behavior_logs` 和 `recommendation_logs` 是推荐系统闭环设计的核心基础表。前者记录用户在平台上的行为信号，后者记录推荐曝光和点击反馈，便于后续进行效果分析和策略优化。

## 6. 基于内容推荐的核心设计与实现

## 6.1 推荐模块设计目标

推荐模块的设计目标主要包括以下几个方面：

1. 提升首页内容分发质量，使不同用户看到更适合自己的校园内容。
2. 兼顾内容热度、内容时效和内容质量，避免推荐结果过于单一。
3. 将用户兴趣标签、关注关系和近期行为纳入推荐依据，增强个性化程度。
4. 形成“曝光 -> 点击 -> 浏览 -> 行为沉淀 -> 再推荐”的数据闭环。
5. 为管理员提供可视化推荐效果分析能力，便于调整推荐策略。

## 6.2 推荐数据基础

推荐功能基于以下几类数据共同完成：

- 帖子内容数据：标题、正文、发布时间、浏览量、点赞数、评论数、收藏数、质量分、置顶/精选状态
- 标签数据：帖子标签、用户标签
- 社交关系数据：用户关注关系
- 行为数据：浏览、点击、分享等行为日志
- 推荐日志：推荐曝光记录与点击记录
- 配置数据：推荐开关与权重参数

推荐服务在后端由 `RecommendationService` 和 `RecommendationServiceImpl` 实现，负责完成推荐召回、排序、记录和统计。

## 6.3 推荐策略实现思路

当前系统采用基于规则加权的内容推荐方案，兼顾实现难度、可解释性和工程落地性。推荐流程如下：

1. 从数据库中读取所有审核通过的内容。
2. 获取当前用户的兴趣标签数据。
3. 读取用户最近的行为日志，根据近期行为进一步增强标签偏好权重。
4. 获取该用户的关注对象列表。
5. 对每一条候选内容分别计算热度分、时效分、质量分、标签匹配分、关注关系分。
6. 按后台配置的权重进行加权求和。
7. 对置顶内容和精选内容给予额外加分。
8. 按最终得分排序并分页返回。
9. 将当前页推荐结果写入推荐日志表，记录曝光数据。

这种方案比单纯按时间排序或热度排序更符合校园内容平台的实际需求，能够在“热门内容”和“个性偏好”之间取得较好的平衡。

## 6.4 推荐评分机制

系统中的推荐评分主要由以下部分构成。

### 6.4.1 热度分

热度分依据帖子已有互动数据计算，主要考虑：

- 点赞数
- 评论数
- 收藏数
- 浏览数

系统通过对不同互动赋予不同权重，构建热度值。评论、点赞、收藏通常代表更强的内容质量与互动意愿，因此权重高于纯浏览量。

### 6.4.2 时效分

时效分反映内容新鲜度。系统通过内容发布时间与当前时间的间隔计算衰减值，使较新的内容在推荐中获得一定优势，避免推荐列表长期被旧内容占据。

### 6.4.3 质量分

质量分来自帖子本身的质量字段，用于承接运营侧或系统侧对内容质量的额外评价，使优质内容在分发中更容易获得机会。

### 6.4.4 标签匹配分

标签匹配分用于表示内容主题与用户兴趣的接近程度。系统先读取用户在 `user_tags` 中配置的兴趣标签，再读取最近行为日志中关联的帖子标签，对用户偏好进行动态增强。如果某条帖子与用户偏好标签重合较多，则该内容得到更高推荐分。

### 6.4.5 关注关系分

如果帖子作者是当前用户关注的人，系统会给予一定加分，使用户在推荐流中也能看到更多来自熟人或关注对象的内容。

### 6.4.6 运营加权与内容加成

后台可通过推荐配置面板控制以下参数：

- 热门流开关
- 关注流开关
- 兴趣标签开关
- 热度权重
- 时效权重
- 质量权重
- 兴趣权重
- 关注权重

同时，对于被设置为“置顶”或“精选”的内容，系统在最终分值中给予额外加成，体现运营干预能力。

## 6.5 推荐闭环实现

为了避免推荐只停留在“静态排序”层面，系统构建了完整的推荐闭环。

### 6.5.1 推荐曝光记录

当用户请求推荐流接口 `/api/posts/recommend` 时，系统除了返回推荐结果，还会将当前返回的每一条内容写入 `recommendation_logs` 表，记录：

- 用户 ID
- 帖子 ID
- 推荐场景
- 推荐分值
- 排名位置
- 是否点击
- 创建时间

这样即可形成曝光数据基础。

### 6.5.2 推荐点击反馈

用户在首页点击推荐卡片、焦点内容或右侧推荐项时，前端会调用 `/api/recommendation-logs/click` 接口，将点击反馈回传后端。系统会将对应推荐日志标记为已点击，若找不到历史曝光记录，则创建兜底点击记录。

### 6.5.3 浏览行为沉淀

用户进入帖子详情页时，系统会：

- 增加帖子浏览量
- 记录一次浏览行为日志

如果该点击来自推荐流，则推荐点击日志也会同步沉淀。这样可以逐步形成用户的行为画像，为后续推荐提供更准确的兴趣依据。

### 6.5.4 推荐效果统计

管理员可通过 `/api/admin/recommend/overview` 查看推荐效果，包括：

- 曝光总数
- 点击总数
- 综合点击率 CTR
- 不同场景的曝光与点击表现
- 高曝光内容排行

这使得推荐系统具备了可观测性，能够支撑后续论文中的效果分析和系统评估部分。

## 7. 前端页面实现说明

## 7.1 用户端页面实现

### 7.1.1 登录注册页

登录注册页用于完成账号接入，包含表单校验、登录状态保存和角色跳转逻辑。

### 7.1.2 首页

首页是推荐能力最直接的展示页面，主要包括：

- 推荐/关注/热议/最新/话题切换 Tab
- 搜索区域
- 焦点内容卡片
- 推荐内容瀑布式卡片流
- 侧边推荐列表
- 热门话题区域
- 快捷入口与消息提醒

首页内容组件在设计上强调高频浏览体验，使用户能够更快速发现感兴趣的校园内容。

### 7.1.3 内容详情页

详情页负责展示完整帖子内容、互动统计与关联内容，是浏览行为沉淀的关键入口。

### 7.1.4 个人中心页

个人中心页展示用户个人资料、发布内容和兴趣偏好，体现平台的用户画像管理能力。

### 7.1.5 互动中心页

互动中心页负责汇总系统通知、社交提醒和举报进度等信息，提升用户对平台动态的感知能力。

## 7.2 管理员端页面实现

管理员端采用现代化后台布局，左侧菜单统一提供以下入口：

- 数据统计
- 控制总览
- 内容审核
- 举报处理
- 内容运营
- 用户治理
- 标签体系
- 公告中心
- 推荐策略

其中“推荐策略”页面除了权重配置外，还加入了推荐效果看板，使后台不再只是配置入口，而是同时具备推荐分析能力。

## 8. 系统关键实现特点

### 8.1 前后端分离，结构清晰

前端专注交互体验与页面组织，后端专注业务逻辑与数据服务，模块边界明确，便于后续扩展和维护。

### 8.2 推荐逻辑可解释

相较于完全黑盒的模型推荐方式，当前系统采用可解释的加权评分方案，便于毕业设计阶段展示推荐依据、评分构成和策略调优思路。

### 8.3 推荐链路完整

系统不只是实现了一个推荐接口，而是完整实现了：

- 推荐候选召回
- 推荐排序计算
- 推荐曝光记录
- 推荐点击反馈
- 行为日志沉淀
- 后台推荐效果分析

这使系统具备了实际平台化运行的基本能力。

### 8.4 管理与运营联动

后台运营配置与推荐模块直接联动。管理员既可以配置标签和公告，也可以调整推荐权重、查看推荐效果、设置置顶和精选内容，从而形成内容治理与分发优化的统一工作流。

## 9. 系统运行流程说明

系统的典型业务流程如下：

1. 用户注册并登录平台。
2. 用户完善个人资料和兴趣标签。
3. 用户浏览首页推荐内容、话题内容或关注内容。
4. 用户进行点赞、评论、收藏、分享、关注等互动。
5. 后端记录用户行为和推荐反馈。
6. 推荐服务根据行为、标签和社交关系动态更新推荐结果。
7. 管理员在后台进行内容审核、用户治理和推荐策略配置。
8. 管理员通过推荐效果看板查看曝光、点击和 CTR 等指标。

## 10. 总结

本系统围绕“校园社交 + 内容推荐”这一主题，完成了从普通社交平台功能到个性化分发能力的完整实现。在基础社交能力方面，系统支持注册登录、内容发布、内容浏览、互动交流、关注关系、消息通知和后台治理；在推荐能力方面，系统基于用户标签、近期行为、关注关系、内容热度、内容时效和内容质量构建了可解释的推荐策略，并进一步实现了曝光记录、点击反馈、浏览沉淀和推荐效果统计分析。

从工程实现角度看，该系统已经具备一个小型校园内容平台的核心能力；从毕业设计角度看，该系统不仅实现了完整业务功能，也突出了“推荐系统如何在真实社交平台中落地”的研究与实践价值，可作为“基于内容推荐的校园社交平台设计与实现”这一课题的系统实现部分正文基础。
